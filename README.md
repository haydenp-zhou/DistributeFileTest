# 分布式文件系统并发测试工具

用于验证分布式文件系统（NFS、Lustre、GPFS、CephFS 等）在多机多进程并发访问时的正确性和性能。

## 特性

- **原子 IO**: 使用 `pread64`/`pwrite64` 保证读写原子性
- **区域隔离**: 每个线程操作不同的 offset 区域，避免冲突
- **数据验证**: 通过 pattern 验证数据完整性
- **多机协调**: 提供 Python 脚本在多台机器上同步启动测试
- **详细统计**: IOPS、吞吐量、错误计数等

## 编译

### 要求
- g++ 7+ 或 clang++ 7+
- Linux 系统（使用 pthread 和 Linux 系统调用）

### 编译命令

```bash
# 使用 make
make

# 或手动编译
g++ -std=c++17 -O2 -o distfs_test distfs_test.cpp -pthread

# 调试版本
make debug  # 生成 distfs_test_debug
```

## 单机测试

### 快速测试

```bash
./quick_test.sh /mnt/nfs/test.dat 100M 4 10
```

参数：文件路径、文件大小、线程数、持续时间（秒）

### 完整参数

```bash
./distfs_test -f /mnt/nfs/test.dat -s 1G -p 8 -d 60 -w 50
```

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `-f, --file` | 测试文件路径 | 必需 |
| `-s, --size` | 文件大小 (支持 K/M/G/T) | 1G |
| `-b, --block` | IO 块大小 | 4K |
| `-p, --parallel` | 线程数 | 4 |
| `-d, --duration` | 测试持续时间（秒） | 60 |
| `-w, --write-ratio` | 写操作比例 0-100 | 50 |
| `--start` | 起始 offset | 0 |
| `--end` | 结束 offset | 文件大小 |
| `--no-prefill` | 跳过预填充 | - |
| `--no-verify` | 跳过数据验证 | - |
| `-v, --verbose` | 详细输出 | - |

## 多机测试

### 使用 Python 启动器（推荐）

**要求**:
- Python 3.6+
- SSH 免密登录到所有目标机器
- 所有机器可访问同一个分布式文件系统路径

**示例**:

```bash
# 3 台机器，每机 8 线程，共测试 10G 文件
python3 distfs_launcher.py \
    --hosts node1,node2,node3 \
    --file /mnt/lustre/test.dat \
    --size 10G \
    --threads 8 \
    --duration 60 \
    --compile \
    --yes
```

启动器会自动：
1. 编译程序
2. 分发到所有主机
3. 计算每个主机的 offset 范围（自动均分）
4. 同步启动测试
5. 收集和汇总结果

### 手动多机测试

如果不想用 Python 脚本，可以手动在每台机器上启动，通过 `--start` 和 `--end` 指定各自的区域：

**机器 A**（操作前 5GB）:
```bash
./distfs_test -f /mnt/nfs/test.dat --start 0 --end 5368709120 -p 16 -d 60
```

**机器 B**（操作 5GB-10GB）:
```bash
./distfs_test -f /mnt/nfs/test.dat --start 5368709120 --end 10737418240 -p 16 -d 60
```

**机器 C**（操作 10GB-15GB）:
```bash
./distfs_test -f /mnt/nfs/test.dat --start 10737418240 --end 16106127360 -p 16 -d 60
```

> 注意：确保所有机器的测试时间一致，可以先用 `date && ./distfs_test ...` 对齐时间。

## 工作原理

### 数据 Pattern

每个写入的 block 包含以下元数据：
- 8 字节：写入时的文件 offset
- 4 字节：写入线程 ID
- 4 字节：写入序列号
- 剩余：随机填充（基于 seed 可重现）

### 验证逻辑

读取时验证：
1. 存储的 offset 是否与实际读取位置匹配
2. offset 是否对齐到 block 边界

### 并发模型

```
文件总大小: 10GB
         ├──────────────┬──────────────┬──────────────┤
机器 A:   [0          - 3.33GB)  线程 0-3
机器 B:   [3.33GB     - 6.66GB)  线程 0-3  
机器 C:   [6.66GB     - 10GB]    线程 0-3
```

每台机器内，线程再次分区：
```
机器 A 区域: [0 - 3.33GB)
            ├──────────┬──────────┬──────────┬──────────┤
线程 0:      [0      - 833MB)
线程 1:      [833MB  - 1.66GB)
线程 2:      [1.66GB - 2.5GB)
线程 3:      [2.5GB  - 3.33GB]
```

## 常见问题

### Q: 测试报错 "pread64 失败: Input/output error"
A: 通常是底层存储出现问题。检查存储系统日志。

### Q: 验证失败数不为零
A: 正常现象。因为读取时数据可能被其他线程覆盖了。重点看"错误数"，应为 0。

### Q: 如何测试 read-after-write 一致性？
A: 先用 100% 写比例跑一轮，再用 0% 写比例（纯读）验证：
```bash
# 第一轮：写入数据
./distfs_test -f /mnt/test.dat -s 1G -p 4 -d 60 -w 100

# 第二轮：验证读取（单线程避免覆盖）
./distfs_test -f /mnt/test.dat -s 1G -p 1 -d 60 -w 0 -v
```

### Q: 支持 macOS 吗？
A: 不完全支持。代码中使用了 Linux 特定的 `pread64`/`pwrite64` 和 `gettid` 系统调用。
